# ELSFM Native — Current Features

Reference inventory of what the Android app (`elsfm-native`) actually does today,
generated by scanning the codebase module-by-module. Backend is the BeMusic/Laravel
API at `https://www.elsfm.com`.

## Architecture at a glance

- **Modules**: `app` (nav host, DI wiring) + 9 `feature:*` modules (auth, library,
  player, downloads, search, discovery, artist, profile, notifications) + 6 `core:*`
  modules (model, network, media, database, designsystem, common)
- **Stack**: Jetpack Compose, Hilt DI, Ktor (kotlinx.serialization), Media3/ExoPlayer,
  Room (offline downloads), MVVM with `StateFlow`-based unidirectional state
- **Auth**: Bearer-token API auth against Laravel Sanctum + `api.access` permission
  gate; Google Sign-In supported alongside email/password

---

## Authentication (`feature:auth`)

- Email/password sign-in, with smart keyboard hints (`.com` suggestion) and
  device password-manager save/autofill on successful login
- Google Sign-In (`GoogleSignInService`), with proper `signOut()` on logout so a
  second account doesn't silently reuse the previous session
- Sign-up (`SignupViewModel` → real `POST auth/register`, distinct from login)
- Forgot-password flow (`PasswordResetViewModel` → real `POST auth/password/email`)
- Logout stops active playback and clears the queue (`PlayerController.stop()`)
  so a track doesn't keep playing under a different signed-in session

## Library (`feature:library`)

- **Tabs**: Songs (liked), Playlists, Albums, Artists, Channels — backed by real
  user-scoped endpoints (`getUserPlaylists`, `getLikedAlbums`, `getLikedArtists`,
  `getChannels`), fetched in parallel
- **Playlists — full CRUD** for the signed-in user's own playlists (owner/editor
  policy on the backend, no special permission needed):
  - Create (`+` icon, name-only dialog)
  - Rename and delete (via a "⋮" menu on the playlist header; hidden for
    curated/channel playlists the user doesn't own)
  - Add/remove tracks (`PlaylistApi.addTrackToPlaylist` / `removeTrackFromPlaylist`
    — the "remove from playlist" action now actually calls the backend instead of
    only updating local UI state)
  - Follow/unfollow API calls exist (`followPlaylist`/`unfollowPlaylist`) but have
    no browse-other-users'-playlists UI yet
- **Playlist track list**: lazy-paginated, with a background loop that keeps
  fetching subsequent pages after the first one loads (so tapping Play right
  after opening a large playlist queues the whole thing, not just page 1),
  stopping cleanly if a page ever fails instead of retrying forever
- **Liked Songs** and **Listening/Play History** screens
- **Album** and **Artist** detail screens (tracklist, follow, play-all)
- Per-track and whole-playlist/album "make available offline" download actions
- Like/unlike toggle on tracks throughout (playlist rows, player, etc.)

## Now Playing / Player (`feature:player`)

- Full playback controls: play/pause, next/previous, shuffle, repeat
  (off → all → one → off), seek bar with live position
- Queue view (bottom sheet), tap-to-jump-to-track
- Track long-press / overflow menu: add to queue, add to library, add to
  playlist (with a playlist picker sheet), share (deep-linkable
  `elsfm.com/track/{id}/{slug}` URL), repost, view lyrics, go to artist/album
- **Download icon** in the now-playing top bar — spinner while downloading,
  checkmark once the current track is saved offline
- **Sleep timer** — 15/30/45/60-minute picker, pauses playback via ExoPlayer
  when it elapses; survives navigating away from the player screen since it
  lives on the app-lifetime `Media3PlayerController` singleton, not the
  screen's ViewModel; cancels cleanly on logout so a stale timer from a
  previous session can't fire against a new one
- Background/lock-screen playback via a `MediaSessionService`
  (`PlaybackService`), with real album-art thumbnails in the system
  notification and tapping the notification opens the app
- Play-history recording per track play (`PlayHistoryApi`)

## Downloads (`feature:downloads`)

- Local offline library: completed downloads listed from Room
  (`DownloadedTrackDao`), with total storage size shown
- Per-track download/delete, backed by the real
  `GET api/v1/tracks/{id}/download` endpoint (mirrors the web PWA's own
  "make available offline" behavior — no bulk backend endpoint, so an
  album/playlist download is the same per-track call looped client-side)
- Offline playback reads the locally saved file instead of streaming

## Search (`feature:search`)

- Search across tracks/albums/artists (real `SearchApi`)

## Discovery (`feature:discovery`)

- Channel browsing (curated content — Kids Zone, Explore More, etc.), channel
  detail screens with nested playlists/albums

## Notifications (`feature:notifications`)

- Notification list screen (`NotificationsApi`)

## Profile / Account (`feature:profile`)

- Account details, theme settings

## Deep linking

- `elsfm.com/track/{id}/{slug}` and `www.elsfm.com/track/{id}/{slug}` open the
  app directly (App Links, `android:autoVerify="true"`) and start playback of
  that track — falls back to a normal (non-verified) intent filter if
  `assetlinks.json` isn't hosted yet

## Backend integration surface (`core:network`)

Real Laravel API clients wired up: `AuthApi`, `UserApi`, `TrackApi`,
`TrackListApi`, `PlaylistApi`, `AlbumApi`, `ArtistApi`, `ChannelApi`,
`SearchApi`, `CommentApi`, `RepostApi`, `LyricsApi`, `NotificationsApi`,
`ProfileApi`, `AccountApi`, `SessionsApi`. All authenticated requests carry a
Bearer token; a transient-401 retry guards against a backend replication-lag
quirk right after fresh login.

---

## Known gaps / not yet built

- No UI for browsing or following *other users'* public playlists (API methods
  exist, no screen)
- No dedicated `Media3PlayerController` unit tests beyond the extracted
  `SleepTimer` class (which is fully tested) — the rest of playback control
  still has no direct test coverage
- Server-side: downloads currently depend on the backend having the
  `api.access` permission granted to the signed-in user's role and a
  syntactically valid `DownloadLocalTrackController.php` deployed — both were
  found broken on production during this engagement and fixes were written
  locally but need deploying
